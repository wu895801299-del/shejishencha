新手版 · Comate 在 fe-aiop 怎么干活

三句话理解 Skills 运行机制

① 你提需求 → ② Comate 自动加载这个仓库的规则 → ③ 命中专业任务时调用对应 Skill 收尾。整页围绕这三句话展开，从浅到深，可按顺序读完。

先看 3 个核心概念 再看 5 步主流程 动手玩联动阅读器

先认识三个名词

  • 规则（Rules）放在仓库里的"自动加载说明书"，告诉 Agent 该怎么做。
  • 技能（Skill）专业任务的工作流，比如 code-validation、visual-verify。
  • 闸门（Gate）必须通过的检查点，没过就不能交付。

第一步：3 个核心概念

整套机制其实只有三块拼图。看懂这一节，下面所有内容都会更顺。

01

规则文件

放在 .comate/rules/ 下的 .mdr 文件。带 alwaysApply: true 的会在每次会话自动加载，相当于"团队默认共识"。

  • base.mdr：协作总闸（启动、UI、闸门）
  • 01-project-introduce：仓库地图
  • 02-programming-style：代码风格
02

Skill

每个 Skill = 一个独立的"专业子流程 + 工具集 + 规则"。任务命中描述关键词时，Agent 必须调用 Skill，而不是凭经验自己实现。

  • code-validation：CR 风格代码审查
  • visual-verify：UI 真机视觉验证
  • auto-commit：智能 commit 提交
03

Gate（闸门）

由 base.mdr 强制规定：什么任务必须经过哪个 Skill 才能收尾。Gate 没过 = 任务没完成，不能跳过、不能用 lint/build 替代。

  • 代码改动 → 必走 code-validation
  • UI 改动 → 必走 visual-verify
  • 提交代码 → 必走 auto-commit

第三步：两道必经闸门

大部分任务都会撞到这两道。看懂触发条件和动作，就知道为什么 Comate 偶尔会"突然多一步"。

code-validation

代码闸门

什么时候触发：任何代码改动后必走，包括新增、修改、重构。

它做什么：用 common/RULES.md + fe-aiop/RULES.md 对改动行做 CR 风格审查，违规必修。

不能用什么替代：本地 lint、build、人工"我看了一遍"都不算。

visual-verify

视觉闸门（仅 UI 任务）

什么时候触发：布局、样式、颜色、间距、交互或响应式有变更时。

它做什么：连真实 Chrome（CDP localhost:9222），跑 DOM 断言 + 截图，留可追溯证据。

不能用什么替代：不能"我觉得对就行"，也不能跳过 CDP 直接想象结果。

什么是 CDP localhost:9222？

CDP（Chrome DevTools Protocol）是 Chrome 内置的远程控制接口，通过 9222 端口的 WebSocket 对外暴露。任何程序连上后可以：

  • 查询 DOM 结构 / 检查元素尺寸与位置
  • 截图（真实渲染结果，非模拟）
  • 在页面上下文中执行 JavaScript
  • 模拟点击、填写表单、页面导航
  • 监听 console.error / console.warn 输出

F12 开发者工具本身就是 CDP 的内置客户端——visual-verify 通过同一接口把 F12 能做的操作全部自动化。9222 是 Chrome 的约定默认端口，可通过环境变量 CDP_HOST=localhost:xxxx 覆盖。

## Basic
When implementing work in `fe-aiop`, keep only constraints that are directly tied to this repository:

1. Understand and reuse the existing page patterns, directory structure, shared components, and state organization before introducing anything new.
2. Prefer the existing organization already used in `src/pages`, `src/modules`, `src/components`, and `src/proComponents`.
3. When adding new capabilities, keep directory placement, naming, state management, and UI choices consistent with the current codebase.
4. When using shared components, refer to `.comate/components/` for component capability documentation (props, usage patterns, and constraints).

## Agent dev startup workflow

When an agent needs to start local `fe-aiop`, use the repository workflow instead of guessing:

1. Before starting, confirm `Node.js >= 24.4.1 < 25` (recommended `24.4.1`), `pnpm >= 10.13.1`, and install dependencies with `pnpm install`.
2. Always prefer `sh dev.sh` over a direct `pnpm start`, because it injects required env vars such as `HOST`, `PROXY_TARGET`, `PROXY`, and `PROXY_TYPE`.
3. Use the exact startup command for the target environment:
   - Default online proxy: `sh dev.sh`
   - Offline XSTP proxy: `sh dev.sh xstp-xxx`
4. For offline joint debugging with `xstp-xxx`, follow `xstp-config.js` exactly:
   - If the user has not provided a usable XSTP name, ask for the target offline environment first.
   - Add or confirm a config item whose `name` starts with `xstp-`, with `hostname`, `serverPort`, `fsrPort`, and optional `eos`.
   - `dev.sh xstp-xxx` passes `PROXY=xstp-xxx` and `PROXY_TYPE=dev`; `xstp-config.js` then resolves the config and starts `pnpm start` with `PROXY_TARGET` from `eos` or `hostname:serverPort`.
   - If the matching `name` does not exist, startup exits instead of falling back, so verify the exact name before recommending the command.
   - After the config is ready, start it with the matching command, for example `sh dev.sh xstp-zjh`.
5. Before asking the user to verify pages locally, remind them of the environment-specific setup:
   - Default online proxy: open `http://imis.baijiahao.baidu.com/bjhmis`, copy the required login cookie such as `PHPSESSID` from `Application -> Cookies`, and write the same cookie into the local debugging domain.
   - `tiance`: add host mapping `127.0.0.1 dev.tiance.baidu-int.com`, open `https://tiance.baidu-int.com/taceo`, and copy `UUAP_S_TOKEN` into that domain's local cookie storage.
   - `mock`: use `localhost`; if cross-origin restrictions appear, use a Chrome profile with cross-origin disabled.
   - Offline `xstp-xxx`: make sure the target environment itself is reachable.
6. If the task only needs code changes or static review, do not start the dev server.
7. Never guess the local access URL. Read the actual Vite output after startup and report only the printed URL.
8. After startup, always return the actual access URL, and also provide these full page URLs based on that origin:
   - Tianmu page: `<access-origin>/aixop/author/manage/all`
   - IMIS page: `<access-origin>/aixop/imis/`
   - Tianyan page: `<access-origin>/aixam/new`

## UI design rule loading

When a task involves any UI/UX visual changes — including layout updates, style adjustments, component appearance, spacing, colors, typography, responsiveness, or interaction behavior — the agent MUST read and follow `05-ui-design.md` before making changes.

## UI interaction constraints

These rules apply globally across all pages in `fe-aiop`, regardless of component type or context:

### Delete action color

**Delete triggers must never use red / danger color.** This applies to:
- Table row operation text links (e.g. "删除")
- Toolbar / action-bar icon or text buttons
- Any other inline delete entry points

Use the same primary color (`#3855d5`) as other operations. The danger color (`#f5222d`) is reserved exclusively for the **confirm button inside a delete confirmation modal** (e.g. `okButtonProps={{danger: true}}`), not for the trigger that opens it.

## Code validation gate

For any task that changes code, treat code validation as a blocking post-implementation gate.

After completing each implementation batch, MUST activate the `code-validation` skill in a standalone subagent before running final validation, reporting completion, or starting an unrelated implementation batch.

The standalone subagent must load the applicable repository rules, review the current changed lines as a CR-style review, and report violations. Fix any reported violations before continuing.

Do not defer code validation until after the final response. Do not treat local build, lint, or tests as a replacement for `code-validation`.

## Visual verification gate

When the task is UI-related:

1. Activate the `visual-verify` skill immediately after stating the UI determination.
2. Verify Chrome DevTools Protocol is available at `localhost:9222` before any browser automation.
3. Stop the task and ask the user to start Chrome with remote debugging if `http://localhost:9222/json/version` does not return a valid Chrome/CDP response.
4. Do not simulate visual verification without a real CDP connection.

# fe-aiop Project Overview

## Project Summary

- A React + TypeScript project for PC-side admin pages.
- The repository is built around existing page templates, business modules, and shared components.

## Core Tech Stack

Use `package.json` as the source of truth for exact versions:
- React 18.2.0 + TypeScript 5.6.2
- Vite 5.3.2
- Jotai 2.6.4
- antd 5.20.2
- React Router DOM 6.22.3
- Less + CSS Modules

## Main Directories

```text
fe-aiop/src/
├── api/             # API definitions and request wrappers
├── assets/          # Static assets
├── atoms/           # Global Jotai state
├── components/      # Shared base components
├── dicts/           # Dictionaries and configs
├── hooks/           # Shared hooks
├── interface/       # Type definitions
├── layout/          # Layout components
├── modules/         # Business reusable modules
├── pages/           # Business pages
├── proComponents/   # Page-level abstractions and schema/proPage capabilities
├── query/           # react-query related wrappers
├── routes/          # Route configuration
├── styles/          # Global styles
└── utils/           # Utility functions
```

## File Placement Rules

1. New files must stay inside the existing directory system and match the current business layering.
2. Components, utilities, and atoms used by only one business page should live inside that page's directory.
3. Shared utility functions belong in `src/utils/`.
4. Shared API wrappers belong in `src/api/`, while page-private API code should stay close to the business directory.
5. Check `src/proComponents/` before building a new reusable page abstraction.

# fe-aiop Coding Style Constraints

## Basic Constraints

1. Keep a single file within about 300 lines when possible; split by responsibility when it grows beyond that.
2. Keep each function focused on a single responsibility, and prefer separating business logic, render assembly, and event handling.
3. New code must match the existing directory structure, naming style, and implementation patterns already used in this repository.

## Naming Conventions

1. React component files use `PascalCase`, consistent with existing files in `src/components`, `src/modules`, and `src/pages`.
2. Utility files use `camelCase`, consistent with `src/utils` and page-local `utils` files.
3. Event handlers should prefer the `handleXxx` naming pattern.
4. Internal helper functions may continue using the existing `_xxx` pattern when it helps distinguish them from exported APIs.

## Implementation Guidance

1. Reuse existing components, hooks, and utilities before adding new ones.
2. Move complex business logic into focused modules instead of letting page entry files grow too large.

# fe-aiop Validation Rules

---

## Rule 1 — UI Components: `@baidu/aiop-ui` First

**Severity**: Error

**`@baidu/aiop-ui` components (must import from here):**
`Button`, `Modal`, `Drawer`, `Tooltip`, `Spin`, `Tabs`, `Alert`, `Select`, `Input`, `Tag`, `DatePicker`, `Checkbox`, `Popconfirm`, `Pagination`, `ConfigProvider`

**Allowed from `antd` directly (not in aiop-ui):**
`Form`, `Row`, `Col`, `Space`, `Flex`, `Steps`, `Cascader`, `InputNumber`, `Divider`

**Icons:** use `@baidu/aiop-icon`, never `@ant-design/icons`.

```diff
- import { Button, Modal } from 'antd';
+ import { Button, Modal } from '@baidu/aiop-ui';

- import { PlusOutlined } from '@ant-design/icons';
+ import { PlusIcon } from '@baidu/aiop-icon';
```

---

## Rule 2 — Styling: Tailwind + `.module.less`

**Severity**: Error

- Tailwind utilities for layout/spacing/color/typography.
- `.module.less` for component-scoped complex styles.
- Never `.module.css` (use `.module.less`).
- No CSS-in-JS (except `antd-style` for theme config).
- No inline `style` objects for static layout/design.

---

## Rule 3 — Data Fetching: `useRequest` / `atomWithQuery`

**Severity**: Error

**Allowed:**
- `useRequest` from `@/hooks/useRequest`
- `axiosRequest` from `@/hooks/useRequest` (imperative calls)
- `atomWithQuery` / `atomWithMutation` from `jotai-tanstack-query`

**Forbidden:**
- `import { useQuery } from '@tanstack/react-query'`
- `import { useDataQuery } from '@baidu/cheetah-hook'`
- `import useSWR from 'swr'`
- `useEffect` + raw `axios`/`fetch` for server-state loading

---

## Rule 4 — API Definitions: Centralized in `src/api/`

**Severity**: Warning

All API URLs must be defined as constants in `src/api/<domain>.ts`. No raw URL path strings (`/gomis/`, `/bjhmis/`, `/sarchmis/`, `/supplier/`) in component, hook, or atom files.

```tsx
// src/api/cockpit.ts
const cockpitApi = {
    log_api: '/gomis/clevermis/cockpit/op_logs',
};
export default cockpitApi;
```

---

## Rule 5 — MPA Sub-App Boundary

**Severity**: Error

6 isolated sub-apps: `tianmu`, `imis`, `qinling`, `supplier`, `tiance`, `tianyan`. Cross-app page imports are forbidden.

- Forbidden: `src/pages/<appA>/` importing from `src/pages/<appB>/`
- Sharing must go through: `src/hooks/`, `src/api/`, `src/atoms/`, `src/proComponents/`, `src/modules/`, `src/utils/`, `src/components/`

---

## Rule 6 — proComponents: PageKey Registration

**Severity**: Error

`FilterPage` and `FormilyPage` from `src/proComponents/` require a `pageKey` registered in `ProPageKey`. Unregistered keys cause runtime errors (`GlobalPageDataMap` cannot resolve the atom).

Flag when:
- `pageKey` is a string literal not in `ProPageKey`
- New `FilterPage`/`FormilyPage` added without a `ProPageKey` entry

---

## Rule 7 — Jotai Async Atoms: `loadable()` Wrapper

**Severity**: Warning

Async atoms consumed in render must use `loadable()` from `jotai/utils`. Access pattern: `{ state, data, error }`.

Flag when:
- Async atom exported without a `loadable()` wrapper
- Component reads async atom directly via `useAtomValue` without `loadable`

## Rule 8 — Route: Lazy Loading Required

**Severity**: Error

Route files (`src/routes/*Routes.tsx`) must use `React.lazy()` for page components. The `element` field in `PageRouteType` must be a `LazyExoticComponent`.

Flag when:
- Static `import` of a page component in a route file
- `element` value not wrapped in `lazy()`

# Common Validation Rules

> These rules apply to **all repositories**, loaded alongside any repo-specific rules.
> All findings here are **Warning** severity unless a repo-specific rule escalates the same topic to Error.

---

## TypeScript

### Avoid `any` type

`any` silences the type checker and hides real bugs. Use `unknown` with narrowing, or define an explicit interface.

```ts
// ❌ any hides type errors
function process(data: any) { ... }

// ✅ explicit type
function process(data: UserPayload) { ... }

// ✅ unknown + narrowing
function process(data: unknown) {
  if (isUserPayload(data)) { ... }
}
```

### No `console.log` in production paths

Remove debug logs before merging. Use a structured logger (e.g., `logger.info(...)`) if runtime logging is genuinely needed.

---

## React

### `useEffect` cleanup

Every effect that registers a subscription, event listener, or timer must return a cleanup function. Missing cleanup causes memory leaks on unmount.

```ts
// ❌ missing cleanup — memory leak on unmount
useEffect(() => {
  window.addEventListener('resize', handler);
}, []);

// ✅ cleanup returned
useEffect(() => {
  window.addEventListener('resize', handler);
  return () => window.removeEventListener('resize', handler);
}, []);
```

### Single primary export per component file

A file named `Button.tsx` should export `Button` as its primary export. Utility helpers should be unexported or moved to a shared utilities module. This keeps imports predictable and avoids accidental re-exports.
